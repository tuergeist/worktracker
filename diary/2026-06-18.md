## Putting session redesign

The previous session entry UI had 10 individual ball buttons — one per putt in the hole. It felt cluttered and was slow to use. Replaced it with **4 aggregate counters**:

```
1-Putt | 2-Putt | 3-Putt | 4+-Putt
```

User enters totals after roughly every hole/round of putts instead of clicking per ball. On save, the frontend reconstructs the per-ball array for the API (e.g. "3 two-putts" → `[2, 2, 2]`) so the backend contract is unchanged.

Also fixed a CSS bug in the process: `.ball` buttons had `color: white` from a global rule that wasn't being overridden, making text invisible on the white button background. One-line fix.

The aggregate approach was the right call. Faster, less error-prone, works with a glove on.

## Mobile-first redesign — decision session

Started a bmad-ux session to plan a mobile-first redesign of the whole app. The app works but it's designed for a laptop, not a phone held in one hand on a putting green.

Key things locked in (full list in `.decision-log.md`):
- Light theme only in v1 — outdoor readability comes first
- Bottom tab nav (Putten / Range) — thumb zone
- One-handed usage is the design constraint, not a nice-to-have
- 48dp touch targets minimum
- Stats are secondary; counter increment is the primary action

The mobile redesign is the next architectural change to the frontend. No code yet, just the design decisions locked.
