## Putting session redesign

The previous session entry UI had 10 individual ball buttons — one per putt in the hole. It felt cluttered and was slow to use. Replaced it with **4 aggregate counters**:

```
1-Putt | 2-Putt | 3-Putt | 4+-Putt
```

User enters totals after roughly every hole/round of putts instead of clicking per ball. On save, the frontend reconstructs the per-ball array for the API (e.g. "3 two-putts" → `[2, 2, 2]`) so the backend contract is unchanged.

Also fixed a CSS bug in the process: `.ball` buttons had `color: white` from a global rule that wasn't being overridden, making text invisible on the white button background. One-line fix.

The aggregate approach was the right call. Faster, less error-prone, works with a glove on.

## Foto-Auswertung: putt-analyzer (`tools/putt-analyzer/`)

New side tool: analyse a photo of the green (drone shots, 4032x2268). Started as
"count balls + distance to hole", but the **real purpose surfaced late**: it's not
counting — it's **putt dispersion relative to the putter→hole line**. The putter
lies along the line balls came from; decompose each ball into **short/long** (along
the axis) and **left/right** (perpendicular), bucket by quality (<=1m good / 1-3m bad
/ >3m rubbish), and surface the **systematic miss** (e.g. "tends short + right").
Core of putting = hole it or land near it. This reframing is what the tool should do.

Key decisions/discoveries:
- **Scale = putter length** (lies flat → ground scale). Verified manually on putt1:
  10 balls all <1m, putter ~1060px → ~1230 px/m, matches the 108mm-hole cross-check.
- **Pure CV fails** on real photos: tilted putters/flags (not axis-aligned), two flags,
  shadow edges, a shoe at the frame edge. RANSAC/directional-morphology unreliable.
- **Pure VLM coords too imprecise** — Mistral/Pixtral hallucinate grids; even as a rough
  seed they land >700px off. Claude **Haiku too weak**, Sonnet over-counts; **only Opus**
  localises hole+putter reliably (~1ct/photo).
- **Architecture: "VLM rough + CV fine"** (Opus default, provider swappable). VLM gives
  semantics (active hole among multiple flags, putter ends, balls-in-hole), CV does the
  precise measurement (white blobs, cup ellipse). Balls = all CV blobs minus in-hole;
  far blobs kept only if the VLM saw a ball there (drops shoe, keeps long putts).
- **Cup centre from the widest rows** — the thin flagstick kept dragging the centroid up
  ("hole detected too high", twice). Fixed.
- **Left/right calibration** is a y-down sign trap: player's right = (u_y, -u_x) with
  u = hole→grip. Locked in a unit test.
- python-decouple for keys (.env), `--save-dir` dumps image+label JSON → dataset to
  **train our own model later** and drop the VLM step (user's idea; needs consent).

Known limits (left for later, documented): ball ~0.1m at the cup vs holed = product
threshold; second flag/hole can leak in as a ball (Opus lists it too); ball count ±1.
5 test images run end-to-end; 11 tests green (+1 live-opt-in). Not committed.
