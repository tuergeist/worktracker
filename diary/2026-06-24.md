## Dev-Login (User-Idee, fürs automatisierte Testen)
- `GET /api/auth/dev-login` — passwortloser Login, **nur** wenn `DEV_LOGIN` env gesetzt (Compose: `DEV_LOGIN=1`, k8s: NIE). Legt `dev@scratchlab.app` an, nutzt `auth_backend.login` → Cookie + 302.
- Frontend: „Dev-Login"-Button im Login-Screen, sichtbar nur auf `localhost`/`127.0.0.1`.
- **Warum nötig**: headless kann man Google-OAuth nicht durchspielen, und das browse-Tool kann den **HttpOnly**-Auth-Cookie nicht injizieren (JWT minten + `cookies set` schlug fehl → 401). Mit dev-login navigiert man einfach auf den Endpoint → Cookie gesetzt → eingeloggt.

## Stolperfallen
- **email-validator lehnt reservierte TLDs ab**: `dev@localhost` (kein Punkt), `dev@…​.test`, `dev@example.com` → alle „special-use/reserved" → `UserRead` (EmailStr) wirft 500 auf `/users/me`. Lösung: echte Domain `dev@scratchlab.app`. (Andere Endpoints gingen, weil nur `/users/me` den User serialisiert.)
- **Putt-Stats zeigten „null"** bei 0 Sessions (aggregate_stats → None → ins Card-Markup). Fix: leerer Zustand wenn `stats.sessions===0` (wie Range es schon machte).
- Test-Helfer: `curl -c jar /api/auth/dev-login` → Cookie-Jar → `curl -b jar /api/...` für API-Tests; browse: `goto /api/auth/dev-login` setzt den Cookie für die UI.
