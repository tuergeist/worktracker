# Decision Log — Golf Training PWA

## Session: 2026-06-18 — Mobile-First UX Redesign

**Project:** Golf Training Tracker (Putten + Range)  
**Context:** App currently functions but is designed desktop-first. Starting a bmad-ux session to redesign for outdoor mobile use.

---

### Decisions

#### 1. Mobile-First Redesign
The existing app is oriented around desktop layouts. All future UI decisions start from the smallest screen (375px wide, one-handed grip) and scale up. Desktop is secondary.

#### 2. Light Theme — No Dark Mode in v1
Golf is played outdoors in daylight. A light theme is essential for screen readability in direct sunlight. Dark mode is deferred — not a v1 concern.

#### 3. Primary Accent Color: Golf Green (`#2f7d32`)
Keep the existing brand color. It reads clearly on a light background and reinforces the sport context. Not up for debate in v1.

#### 4. Bottom Tab Navigation (Putten | Range)
Replace any top-tab or sidebar navigation with a bottom tab bar. Two tabs: **Putten** and **Range**. Bottom placement puts the primary navigation gesture in the thumb zone.

#### 5. Putting — Batch Entry via 4 Aggregate Counters
Rather than 10 individual ball buttons (one per putt), the session uses 4 counters:  
`1-Putt | 2-Putt | 3-Putt | 4+-Putt`  
User taps after roughly every 10 putts. The frontend reconstructs the per-ball array for the API on save. Backend API is unchanged. This is already implemented and confirmed correct.

#### 6. Range — Carry Only, No Drift in v1
Range sessions record carry distance only. Drift (left/right) is not captured in v1. Rationale: users who have Trackman don't need this app. The target user is practicing without launch monitor data; carry is the meaningful number.

#### 7. One-Handed Usage Pattern
All primary actions (start session, increment counters, save) must be reachable with the thumb of one hand. The phone is held in the non-dominant hand; dominant hand holds the club. Design for glanceable, one-tap interactions.

#### 8. Minimum 48dp Touch Targets
All interactive elements (buttons, counters, tabs) must meet the 48dp minimum touch target size. This is non-negotiable for outdoor use with golf gloves on.

#### 9. Photo of Putting Green — Future Feature, Hook into IA Now
Attaching a photo of the green (slope, layout) to a session is a desirable future feature. The information architecture should accommodate this without requiring structural changes later. No implementation in v1, but a placeholder should exist in the session detail model.

#### 10. Exercise/Club Selector — Compact at Top
The exercise or club selector is a utility, not the hero. It lives in a compact form at the top of the session screen. It should not dominate the layout.

#### 11. Stats — Secondary Screen, Not Default Visible
Statistics and trend charts are valuable but not the primary use case during a session. Stats live on their own screen, not surfaced by default on the main session view.

#### 12. Players/Profiles — Compact in Header
Multi-user profile switching is a header-level concern. A compact avatar/name indicator in the header is sufficient. No full-screen profile management on primary flows.
