# 2026-06-29

## Auto-Deploy nach k8s-infra verdrahtet
Frage „wird automatisch deployt?" → nein. Recherchiert wie die anderen Apps deployen.

**Cluster/CD-Muster (k8s-infra, expert-sieve):** **kein ArgoCD** (das nutzt nur k8s-infra-**vsx**). k8s-infra deployt per **CI-Push**: App-Repo baut+pusht Image → `repository_dispatch` (event `deploy-simple`) an `expert-sieve/k8s-infra` → `deploy-simple.yml` macht `kustomize edit set image`, committet, `kubectl apply -k apps/<app>/`. Manueller Tag-Bump = Trigger. (vsx-Apps dagegen: ArgoCD GitOps, automated sync.)

scratchlab war in k8s-infra schon angelegt (`apps/scratchlab/`, in `deploy-simple.yml` registriert, image=`ghcr.io/tuergeist/worktracker`), aber **vom 19.6. → veraltet/kaputt**.

## Gemacht
- **k8s-infra/apps/scratchlab** (commit de55251): `DATABASE_URL` → `postgresql+asyncpg://`; **migrate-initContainer** (`alembic upgrade head`, gleiches Image — kustomize rewritet auch initContainer-Tag); `SESSION_SECRET`-env (required); unbenutztes `GOOGLE_REDIRECT_URI` raus; Ingress `proxy-body-size: 16m`. `kubectl kustomize` rendert sauber (verifiziert).
- **worktracker docker.yml** (commit 68e263f): Tag `sha-long` → `sha-short`; `peter-evans/repository-dispatch`-Step (master, kein PR) → `deploy-simple` `{"app":"scratchlab","image_tag":"sha-<short>"}`.

## Blocker (user-only, kann ich nicht)
1. **`K8S_INFRA_DISPATCH_TOKEN`** als GitHub-Secret im worktracker-Repo — CI-Lauf 68e263f failte genau hier (Secret NOT SET). Build+Push liefen durch. Token: dedizierter PAT mit Dispatch-Recht auf expert-sieve/k8s-infra (wie eink/hyperliquid). `gh auth token` wäre zu breit → bewusst nicht gesetzt.
2. **`SESSION_SECRET`** ins Live-Secret `scratchlab-secrets` — **vor** dem ersten Deploy, sonst Pod-Start-Fehler (required secretKeyRef). `kubectl -n scratchlab patch secret … SESSION_SECRET=$(openssl rand -hex 32)`.
3. **Google Console**: Redirect-URI `https://scratchlab.app/api/auth/google/callback` (nicht `/api/auth/callback`).
4. Mein AWS-SSO abgelaufen → kein lokaler Cluster-Zugriff; Deploy läuft eh in CI mit eigenem Kubeconfig.

**Reihenfolge wichtig:** erst (2) SESSION_SECRET ins Live-Secret, dann (1) Token → nächster Push deployt. Sonst crashloopt der Pod.
