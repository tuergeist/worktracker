# 2026-06-22

Google-Login live gebracht + Alembic-Migrationen + Onboarding-Intro.

## Login zum Laufen gebracht (Debug-Kette)
Nach dem fastapi-users-Umbau drei Fehler nacheinander beim echten Login:
1. **People API**: httpx-oauth `GoogleOAuth2.get_id_email` ruft `people.googleapis.com` → muss in der Google-Cloud-Konsole aktiviert sein (`GetIdEmailError`). Mein früheres „keine API nötig" war falsch — für httpx-oauth ist die People API Pflicht.
2. **oauth_accounts.user_id fehlte**: `SQLAlchemyBaseOAuthAccountTable` setzt FK auf `user.id`, unsere Tabelle heißt `users`. Schon gefixt (declared_attr → `users.id`), aber die Tabelle war aus einem früheren Boot stale.
3. **Stale `is_default` INTEGER** im alten Volume → `IS TRUE must be boolean`. Lösung jedes Mal: `docker compose down -v`. Genau das motivierte Alembic.

## Alembic statt create_all
`Base.metadata.create_all` legt nur fehlende Tabellen an, **altert nie** → bei jedem Schemawechsel Volume-Drop. Jetzt:
- Alembic async-env (asyncpg) mit `pg_advisory_xact_lock` → mehrere Pods serialisieren `upgrade`, kein Race.
- Initial-Migration autogeneriert (`eb9f00d9fd86`).
- App legt keine Tabellen mehr an: lifespan = `wait_for_db` + `seed_defaults`. Schema via `alembic upgrade head`.
- Lauf-Ort: **eigener Schritt** — Compose `migrate`-Service (vor App via `service_completed_successfully`), k8s **initContainer**.
- **Stolperfalle**: migrate-Image muss die Migrationsdatei enthalten — Datei wird per Autogenerate erst nach dem Build erzeugt → danach **neu bauen**, sonst „upgrade" ohne Effekt (alembic_version leer).
- Workflow künftig: `docker compose run --rm -v "$PWD/migrations:/app/migrations" app alembic revision --autogenerate -m "…"` → committen → Deploy upgradet selbst.
